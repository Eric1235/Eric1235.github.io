2月3日 天气:阴

经过我的验证，使用atomatic类，性能会比synchronized关键字的同步快三倍。
关于对全局变量的理解：虽然使用起来方便，但是代码多起来的时候，很可能会出现bug，特别是多线程的时候。
增量式开发：在遇到一个新问题的时候，我之前没有接触过，我应该怎么去逐步解决呢？答案就是要把大问题逐步分解成为小问题。比如先定义出数据结构，去写空的函数，再逐步写出可以用的函数。比如，在做自定义控件的时候，如果不会，我可以先画个背景，然后再逐步去画其他的。

2月24日 天气:阴

今天学习的就是一个购物车的按钮控件。也是照着学习的。学习到的地方，就是要把动画效果分成多个过程。
分析如下：
初始状态，添加状态，减少状态。
四个动画控制器：hint区域展开和缩小，增加和减少动画。彼此之间的执行顺序是有一定的连贯性的。比如在初始化的时候，显示的是加入购物车几个字，点击的时候，执行hint区域减少至消失的动画，在这个动画执行完毕的时候，接着执行减号从右向左弹开的动画。这两者，都是需要一定的渐变梯度去控制。

控制onDraw的状态量：fraction以及各个状态的开关。
由于ondraw是把整个view进行重绘的，这时就要去使用一些变量去控制绘制流程。比如hint区域缩小的时候，rect左边的位置，就是是根据hintanimfraction这个变量来不断去绘制新图形。圆圈滚动的时候也是一样，圆圈的位置变化，也是跟着valueanimator改变梯度，重绘界面。

下面我们来介绍一下android方面的，我之前忽略的可能导致内存泄漏的地方之匿名内部类。

r2多引用了一个MainActivity对象，如果将r2传递给一个异步线程，造成线程的生命周期与activity的不一致，造成内存泄漏。

  • 如果成员变量多是static的话，就会造成内存占用过大。不能在类初始化的时候就初始化静态成员，可以考虑延迟初始化。
  • 远离非静态内部类和匿名类，多用private static class。context要使用application那边来的。
  • 退出activity的时候，多记得去清理集合。如：ArrayList

下面去拿一下别人的总结成果：

  • 构造 Adapter 时，没有使用缓存的 convertView
  • Bitmap 对象不在使用时调用 recycle() 释放内存
  • Context 使用不当造成内存泄露：不要对一个 Activity Context 保持长生命周期的引用。尽量在一切可以使用应用 ApplicationContext 代替 Context 的地方进行替换。
  • 非静态内部类的静态实例容易造成内存泄漏：即一个类中如果你不能够控制它其中内部类的生命周期（譬如Activity中的一些特殊Handler等），则尽量使用静态类和弱引用来处理（譬如ViewRoot的实现）。
  • 警惕线程未终止造成的内存泄露；譬如在 Activity 中关联了一个生命周期超过 Activity 的 Thread，在退出 Activity 时切记结束线程。一个典型的例子就是 HandlerThread 的 run 方法是一个死循环，它不会自己结束，线程的生命周期超过了 Activity 生命周期，我们必须手动在Activity的销毁方法中中调运 thread.getLooper().quit(); 才不会泄露。
  • 对象的注册与反注册没有成对出现造成的内存泄露；譬如注册广播接收器、注册观察者（典型的譬如数据库的监听）等。
  • 创建与关闭没有成对出现造成的泄露；譬如 Cursor 资源必须手动关闭，WebView 必须手动销毁，流等对象必须手动关闭等。
  • 不要在执行频率很高的方法或者循环中创建对象（比如onmeasure），可以使用 HashTable 等创建一组对象容器从容器中取那些对象，而不用每次new与释放。
  • 避免代码设计模式的错误造成内存泄露；譬如循环引用，A持有B，B持有C，C持有A，这样的设计谁都得不到释放。

RN的知识路过：对rn进行热更新技术，就是替换掉它里面的index.android.bundle文件。
热更新思路有两种：分离bundle和打包补丁软件。

  • bundle里存放了RN源码、第三方库代码和业务逻辑代码，其中频繁更新的就只有业务逻辑代码，所以我们将RN源码和第三方库代码打包成一个bundle，业务逻辑打包成一个bundle，热更新的时候就只更新业务逻辑的bundle即可。
  • 我们可以使用bsdiff对比两个版本的bundle文件得到差异文件，也就是“补丁”，客户端下载好补丁文件，将其与本地的bundle进行融合从而得到最新版本的bundle文件。
