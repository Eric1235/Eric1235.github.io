    8月2日 天气:晴

    • java8引入了time包，为的就是解决java之前的时间类库残缺不全的问题。其中最重要的类就是LocalDate，它是表示日期的不可变类型，不包含时间和时区。
    • LocalDate使用单一的历法系统，ISO-8601历法系统是事实上的世界民用历法系统，也就是公历。平年有365天，闰年是366天。闰年的定义是：非世纪年，能被4整除；世纪年能被400整除。为了计算的一致性，公元1年的前一年被当做公元0年，以此类推。

    MySQL的时间类型，以后一般还是倾向于第一种

    • datetime 8 bytes YYYY-MM-DD HH:MM:SS 1000-01-01 00:00:00 ~ 9999-12-31 23:59:59
    • timestamp 4 bytes YYYY-MM-DD HH:MM:SS 1970-01-01 00:00:01 ~ 2038
    • date 3 bytes YYYY-MM-DD 1000-01-01 ~ 9999-12-31

    • year 1 bytes YYYY 1901 ~ 2155

    • 了解老板端的功能架构，画出思维导图。

    • @PathVariable注解，就是处理请求uri的一种方式，在例如@RequestMapping(“/owners/{ownerId}”) 的情况下，方法就可以使用（@PathVariable String ownerId） ;

    • @RequestHeader 注解，可以把Request请求header部分的值绑定到方法的参数上。
    • 缓存策略：对于经常需要去进行缓存的东西，我们要去使用Redis进行缓存起来。下次读取的时候，可以从缓存那里读取，而不用跑到MySQL中，进行读取。

    • url的区分，有api的就是接口api，没有的话，那么就是jsp页面，目前我是这样看见的。

    • @JsonSerialize注解，可以将json数据中的类型进行格式化，比如说加上@JsonSerialize(using = BeStringSerializer.class)，就是要把数据强行转为字符串。还有另外的转换器，比如把时间戳格式化成为时间。

    • 我好像懂了二维码登录是怎么回事了；浏览器获得一个临时 id，通过长连接等待客户端扫描带有此 id 的二维码后，从长连接中获得客户端上报给 server 的帐号信息进行展示。 并在客户端点击确认后，获得服务器授信的令牌，进行随后的信息交互过程。 在超时、网络断开、其他设备上登录后，此前获得的令牌或丢失、或失效，对授权过程形成有效的安全防护。

    • 上传文件弹出路径错误，Unix系统识别不出“//”两个斜杠，所以导致错误。使用工具类去进行消除，就能够解决。

    8月3日 天气:雨

    • 订单的排序，是按照最后的状态修改时间来进行排序，其他的修改都会去忽略。
    • 点击次数刷新，主要是前台通过ajax去访问后台，每一秒主动请求数据，刷新界面。次数没有刷新到，是因为没有被select出来。加入那个条件去进行筛选，就能够得到所有的点击数据。
    • 上传文件的时候，上传失败。查log后才发现是服务器路径配置错误，本来是linux的路径，这里错误配置成为windows的了。其实我在其他地方看过，是可以运行时获取系统类型，然后才去根据不同的系统类型，生成不同的路径。
    • 另外发现的一个问题，上传错误的log，这时会被后面的log去覆盖掉。验证文件的时候，出现了一些log，本来是已经导入数据库了的。但是后面有的一个逻辑，就是比对一个商品，要整个商品的所有图片上传失败什么的，那个才会返回false，此时的error并不是之前的error，而是另外一个error，把之前的会冲掉。解决办法就是:为什么要分两次呢？把这个log一直带着啊，到最后才存进去数据库。
    • 路径错误的判断，有可能是url拼写错误，有可能是文件夹未创建，还有一种可能就是文件夹创建了，但是你没有权限。
    • 有一些地方的异常捕获不足，比如说异步的时候，路径错误了，这个信息并没有返回给前台，前台那里显示了上传成功，实际上是没有成功的。我要做的就是，把这个异常抛到上层，让上层能够看到。
    • 目前还遇到的一个问题，就是当所有图片都上传失败的时候，貌似没有任何提示。我这边就加入了对上传的item进行判断，如果一张图片都没有上传，那么就是更新失败。相当于提了一个新的需求，图片没有的时候，提示全部失败，加入这个判断。

    8月11日 天气:晴

    • model传递java对象到jsp的话，单纯使用js去接收，是不行的。所以，我要传递json数据过去，就好了。

    • 发送一条透传消息给用户。透传消息就是那种带参数的消息，制定一个协议，接收消息以后点击，能够对消息进行处理。
    • 吐槽一下，我看的原型，疏漏还是很多的。今天又看到自己漏了一个功能，得赶紧去补回来。我想要双屏。
    • 默认spring事务只在发生未被捕获的 runtimeexcetpion时才回滚。spring aop 异常捕获原理：被拦截的方法需显式抛出异常，并不能经任何处理，这样aop代理才能捕获到方法的异常，才能进行回滚，默认情况下aop只捕获runtimeexception的异常，但可以通过配置来捕获特定的异常并回滚换句话说在service的方法中不使用try catch 或者在catch中最后加上throw new runtimeexcetpion（），这样程序异常时才能被aop捕获进而回滚
    • 添加好友时候，需要建立两条数据，等于进行双向建立。明白这个业务逻辑了。
    • 数据库的唯一键，是可以制定多个字段为唯一的。
    • 有一种无聊的情况，就是添加好友的时候，自己添加自己。我在我写的接口中，拒绝了这种事情的发生。

    • 分享链接生成的api，写错地方了，现在需要写到h5的项目去。不过，学习到的知识，就是使用短网域名去进行发布，一般其他人就不会猜测到是什么内容。
    • 关于页面可不可以点的问题，就是传递一个布尔参数过去，确定页面可不可以点。spring mvc有一个很好玩的功能，就是可以指定那个参数传或者不传。在不传的时候，就设置一个默认值给他。我这里的jsp适合内部和外部进行使用，内部的时候，需要传递这个布尔值，表示不可以点击。外部页面，微信等用到的时候，就不传这个值了，这样，外部的页面就是可以点击的。
    • jsp中，如何增加一些额外的属性
    • 关于短网域名的问题，不使用明文是为了安全
    • 公司有把定时任务的调度给抽离出来，新开启一个项目去调度redis队列。
    • 表被改过了，插入不了，必须修改字段
